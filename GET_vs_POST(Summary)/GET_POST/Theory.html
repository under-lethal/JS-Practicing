2.Client Side Validation

		Client Computer												Web Server
		(Web Browser)	
	
	Clietn Computer - runnin a WEB Browser.								A web Server somewhere else in the world.
													On the WebServer - there"s a HTML Page with JS code inside it.
													A form Handler, and - the server can CONNECT to a DataBase.
													An HTML file - ALREADY exist on the SERVER - or it could be GENERATED by a PROGRAM (such as PHP) - WHEN REQUESTED.
	the Web Page is REQUESTED by the CLIENT. - Either by typing its URL - (or following a link from another page).
	The PAGE is DELIVERED and RENDERED - by the WEB Browser.
	The user FILLS IN the Form (on the web page) - and JS PROGRAMS (inside the page) - validate the DATA.
	These PROGRAMS are RUNNING by the BROWSER - on the CLIENT Computer. - This is called client-side validation.
	When the validation checks - are SATISFIED - the DATA - are posted - back to the SERVER.
	The form data - are PROCESSED by the form-handler. - This could INVOLVE  (more validation CHECKS) and any number of OTHER processes (including SAVING the DATA to a DATABASE).
	Client Side Validation - can IMPROVE the User Experience (by CATCHING INVALID DATA) before it"s send to the SERVER.
	If all of the VALIDATION (was done by the SERVER ALONE) - the user will EXPERIENCE a DEELAY - before BEING told what they need to CHANGE, in order to put WRONG data RIGHT.
	There should be ALWAYS some security CHECKS on the SERVER - as WELL.
	For example: - If a USER is inputting a new user Name that HAS to be UNIQUE.
	The SERVER side CODE - will need to check if it"s in the DATABase - to see if the USERNAME has already been TAKEN.

	You can make A VALIDATION  - without using / writting JS code.
	HTML5 form elements - have THIS capability built into them.
	HTML Validation - PERFORM better than JavaScript Validation - but it"s LESS FLEXIBLE.
